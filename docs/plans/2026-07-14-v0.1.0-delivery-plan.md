## 1. 目标与停止条件

本轮只交付一个有限目标：让 MemoArk v0.1.0 具备可靠的升级、迁移和失败恢复能力，并形成可重复验证的本地发布候选版本。

满足以下全部条件即停止开发，不再以“顺手优化”为理由继续扩项：

1. #1 SQLite 升级前备份、#2 JSON 导入、#3 保存失败恢复全部完成并通过验收。
2. 已完成的 #4 多标签 AND 筛选无回归。
3. Go、前端测试、lint、build 和必要的 protobuf 检查全部通过。
4. 旧 SQLite 升级、全新安装、Docker 重启、导出再导入、断网保存与重试均有可复核证据。
5. 正式本地容器稳定运行，代码、文档、GitHub issue/milestone 状态一致，工作区干净。
6. 最终只集中推送一次；推送前先完成全部本地验证并确认不会无意义消耗 GitHub Actions 额度。

## 3. 执行约束与防循环规则

1. **单一事实源**：本文件是本轮唯一施工表；只允许更新状态、证据、实际偏差，不临时扩充目标。
2. **阶段封口**：某阶段的验收命令通过并记录证据后即封口；除非后续回归证明它被破坏，否则不再返回润色。
3. **最小改动**：优先复用现有 migrator、Connect RPC、memo export、editor cache 和测试工具，不重写相邻模块。
4. **数据优先**：涉及迁移或正式容器替换前先备份现有 SQLite；任何备份/校验失败都必须在修改源库前终止。
5. **失败收敛**：验证失败时先定位根因，只做一个最小修复再跑最小相关检查；确认是额度、权限、Runner、网络等外部问题后立即停止，不重复 rerun。
6. **提交纪律**：每个阶段可做一个本地里程碑提交；所有阶段通过前不 push、不开 PR、不触发 CI。
7. **兼容约束**：不得降低现有权限校验；不得改变未列入范围的导出格式；数据库结构变化必须覆盖 SQLite/MySQL/PostgreSQL 三套迁移，否则不引入结构变化。
8. **可观测性**：关键路径需要清楚日志/结果摘要，不能只靠“没有报错”判断成功。

## 阶段 A：SQLite 迁移前可验证备份（20%）

**依赖**：阶段 0。
**目标**：仅在 SQLite 存在待执行迁移时，在任何源库改动前生成一致、可校验、可人工恢复的备份。

| ID | 任务 | 约束 | 验收 |
| --- | --- | --- | --- |
| A1 | 在 migrator 中判断“SQLite + 非全新库 + 存在待执行迁移” | 无迁移/全新库不备份；其他数据库不进入此流程 | 分支单元测试 |
| A2 | 使用 SQLite 在线备份能力创建一致快照 | 不直接复制活动 WAL 数据库文件 | 旧版本 fixture 只生成 1 个备份 |
| A3 | 生成含源版本、目标版本、UTC 时间的文件名和日志 | 路径可定位、名称不冲突 | 日志断言 + 文件断言 |
| A4 | 对备份执行 `PRAGMA integrity_check` | 结果不是 `ok` 即失败 | 备份可独立打开且 integrity 为 ok |
| A5 | 失败时中止迁移并保持源库版本/内容不变 | 不允许“备份失败但继续升级” | 不可写目录/注入失败测试 |
| A6 | 写人工恢复说明 | 只覆盖 SQLite 本地恢复 | 文档命令可执行 |

**阶段验证**：

```powershell
go test ./store/...
go test ./...
```

**退出条件**：旧库升级、无迁移、全新库、备份失败四类路径均通过；本地提交并把状态更新为已验证，总进度 45%。

## 阶段 B：空账户导入 MemoArk v1 JSON（25%）

**依赖**：阶段 A；复用现有 `memoark.memo-export` v1。
**目标**：用户可以预检并把受支持的导出字段原子地恢复到空账户，失败不留下半套数据。

| ID | 任务 | 约束 | 验收 |
| --- | --- | --- | --- |
| B1 | 共享导出格式契约并实现严格解析/校验 | 只接受 format=`memoark.memo-export`、version=1 | 错格式/错版本/损坏 JSON 在写入前失败 |
| B2 | 生成导入预览 | 显示正常、归档、无效、未知/不支持内容计数 | fixture 计数准确 |
| B3 | 增加鉴权后的服务端导入接口 | 仅当前用户；账户必须无 memo | 非空账户被拒且零写入 |
| B4 | 在单事务中恢复受支持字段 | content、visibility、pinned、create time、state | 中途注入失败后账户仍为空 |
| B5 | Settings 增加选文件、预览、二次确认和结果摘要 | 明示附件文件/评论/设置不导入 | 可访问、可取消、错误可理解 |
| B6 | 导出→清空测试账户→导入 round-trip | 受支持字段一致、计数一致 | 单元/集成/浏览器测试通过 |

## 阶段 C：保存失败后本地保留与安全重试（20%）

**依赖**：阶段 0；可与导入实现解耦，但在阶段 B 后统一集成。
**目标**：新建/编辑 memo 保存失败时内容留在本地，用户明确看到未送达状态，刷新后能恢复并安全重试且不产生重复 memo。

| ID | 任务 | 约束 | 验收 |
| --- | --- | --- | --- |
| C1 | 扩展 editor cache 为待提交状态机 | 保留现有 v2 cache 的兼容读取或安全迁移 | 旧缓存不丢，状态和时间戳可恢复 |
| C2 | 区分网络/服务不可用与业务校验错误 | 不把权限/校验错误伪装成离线 | UI 文案和测试覆盖两类错误 |
| C3 | 失败时立即 `saveNow` 并显示“已保存在本地，尚未发送”+时间 | 只有服务端成功才清缓存 | 失败后刷新内容仍在 |
| C4 | 提供显式重试 | 防双击；沿用同一编辑目标/客户端操作身份 | 新建/编辑各重试成功一次 |
| C5 | 防止响应丢失造成重复新建 | 不引入完整后台队列 | 重复点击/迟到响应测试只有一个 memo |
| C6 | 增加断网、重载、重试自动化测试 | 覆盖 create 和 edit | 浏览器/组件测试稳定通过 |

**阶段验证**：

## 5. 验证矩阵

| 场景 | 自动验证 | 真实环境验证 | 通过标准 |
| --- | --- | --- | --- |
| SQLite 有待迁移 | Go fixture test | Docker 旧库启动 | 先有 1 个 integrity=ok 备份，再升级 |
| SQLite 无待迁移 | Go test | Docker 重启 | 不新建备份，服务正常 |
| SQLite 备份失败 | 注入失败 test | 只用隔离目录模拟 | 源版本/内容不变，启动清楚报错 |
| JSON 合法导入 | Go/TS round-trip | 浏览器空账户导入 | 支持字段和计数一致 |
| JSON 非法/版本错误 | parser/API test | 浏览器错误展示 | 零写入，错误可理解 |
| 非空账户导入 | API test | 浏览器已有 memo 账户 | 被阻断，原数据不变 |
| 新建 memo 保存失败 | component/e2e | 浏览器断网 | 本地状态可见，刷新可恢复，重试仅 1 条 |
| 编辑 memo 保存失败 | component/e2e | 浏览器断网 | 服务端旧值不被误报为成功，本地稿可恢复 |
| 多标签 AND | 既有 TS tests | 浏览器选择两个标签 | 只显示同时具备两个标签的 memo |
| 集成部署 | 全量命令 | 正式容器启动/重启 | HTTP 200、日志无迁移/导入错误 |
